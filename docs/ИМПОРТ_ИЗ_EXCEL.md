# Импорт коллекции из Excel

Документ описывает идею и варианты реализации функции «загрузить Excel-файл и добавить все монеты в коллекцию». **Реализовывать пока не планируется** — только анализ и план на будущее.

---

## 1. Главная сложность — сопоставление (matching)

В каталоге у монет есть внутренний `id` (например `3637`, `4059`). В Excel пользователь может вести учёт по-разному:

| Что может быть в Excel | Наш каталог | Как сопоставить |
|------------------------|-------------|------------------|
| Номер ЦБ РФ (5111-0001, 3213-0005) | `id` — свой числовой | Нужна таблица mapping catalog_number → coin_id |
| Название («Один червонец Сеятель 1975») | `title` | Нечёткий поиск — ненадёжно, много дублей |
| Год + номинал | `year`, `faceValue` | Очень много монет с одинаковым годом/номиналом |
| Своя нумерация | — | Не совпадёт |

**Вывод:** без единого идентификатора импорт превратится в гадание. Реалистичный путь — либо колонка с нашим `coin_id`, либо колонка с каталожным номером ЦБ РФ (если есть в данных).

---

## 2. Варианты реализации

### Вариант A: Строгий — только колонка с ID

- Формат: колонка A = `coin_id` (наш id: 3637, 4059 и т.д.).
- Плюсы: простой парсинг, надёжный импорт.
- Минусы: пользователь должен знать наши id или скачать шаблон.

### Вариант B: Каталожные номера ЦБ РФ

- Если в данных есть `catalog_number` (5111-0001, 3213-0005):
  - Строим mapping catalog_number → coin_id.
  - Пользователь указывает колонку с каталожным номером.
- Плюсы: нумизматы часто ведут учёт по номерам ЦБ.
- Минусы: нужно проверить, есть ли catalog_number в нашей БД/экспорте.

### Вариант C: Нечёткий поиск по названию

- Excel: название, год, номинал.
- Сервер: fuzzy match по каталогу.
- Плюсы: гибкость формата.
- Минусы: неоднозначности, ошибки, сложная логика.

### Вариант D: Шаблон Excel от нас

- Выдаём шаблон: колонки `coin_id` (или catalog_number), `quantity`, `buy_price`.
- Пользователь заполняет и загружает.
- Плюсы: предсказуемый формат.
- Минусы: нужно адаптировать свою таблицу под наш формат.

## 3. БД и Supabase

**Текущая схема:**

```sql
user_collection (user_id, coin_id) unique(user_id, coin_id)
```

**При импорте:**

- Для каждой монеты: `INSERT` с `ON CONFLICT (user_id, coin_id) DO NOTHING` (игнорировать дубликаты).
- При 500 монетах: лучше один batch insert, чем 500 одиночных запросов.

**Supabase:**

- Бесплатный план: лимиты на запросы в минуту — batch снижает нагрузку.
- RLS: пользователь видит только свои записи.

**Рекомендация:** Supabase RPC `import_collection(coin_ids text[])`, который делает batch insert с `ON CONFLICT DO NOTHING`.

---

## 4. Размер файла и лимиты

| Параметр | Рекомендация | Зачем |
|----------|--------------|-------|
| Макс. строк | 500–1000 | Ограничить нагрузку, UX |
| Макс. размер файла | 1–2 МБ | Защита от DoS |
| Формат | .xlsx (можно .csv) | Парсинг через библиотеку (xlsx, sheetjs) |

- Excel может содержать макросы — **не выполнять**, только читать данные.
- Валидация: проверять типы ячеек, пустые строки.

---

## 5. Где выполнять логику

### Вариант 1: Только на клиенте

- Загружаем файл → парсинг в браузере (xlsx) → массив coin_id → по одному вызову `addToCollection` или batch через API.
- Минусы: при 500 монетах — много запросов или долгий цикл.

### Вариант 2: Edge/Serverless API

- Маршрут `POST /api/import-collection`: принимает файл, парсит, делает batch insert в Supabase.
- Плюсы: контроль нагрузки, валидация на сервере.
- Минусы: проект сейчас статический (`output: 'export'`), API в проде не работает — нужен хостинг с Node (Vercel Functions и т.п.).

### Вариант 3: Гибрид — клиент парсит, Supabase RPC пишет

- Клиент: парсинг Excel → массив `coin_id`.
- Один вызов Supabase RPC `import_collection(coin_ids text[])` — внутри batch insert.
- Плюсы: один запрос в БД.
- Минусы: нужно создать RPC в Supabase.

---

## 6. Вес и лимиты

- Одна строка в `user_collection`: ~50 байт.
- 500 монет: +25 КБ — несущественно.
- В `docs/ТАРИФЫ.md` упомянуты лимиты (например 50 монет на бесплатном). При импорте нужно проверять лимит тарифа (если он реализован).

## 7. UX-поток

1. Страница «Моя коллекция» → кнопка «Импорт из Excel».
2. Выбор файла (или drag-and-drop).
3. Парсинг → превью: «Будет добавлено 47 монет, 3 не найдено в каталоге».
4. Пользователь подтверждает.
5. Выполнение импорта.
6. Отчёт: «Добавлено 45, 2 уже были в коллекции, 3 не найдено».

---

## 8. Шаги для реализации (когда будем делать)

1. Определить источник идентификатора: `coin_id` или `catalog_number` (проверить наличие в данных).
2. Сделать шаблон Excel (пример колонок) и скачивание шаблона.
3. Добавить Supabase RPC `import_collection(coin_ids text[])` или batch insert через клиент.
4. На клиенте: форма загрузки, парсинг (xlsx), вызов RPC.
5. Превью и отчёт по результату.

---

## 9. Минимальный MVP

- Шаблон Excel: одна колонка `coin_id` (наш id).
- Кнопка «Импорт» на странице коллекции.
- Клиент: парсинг → валидация id по `coin-ids.json` → batch insert (RPC или несколько insert батчами).
- Показать: сколько добавлено, сколько не найдено.
