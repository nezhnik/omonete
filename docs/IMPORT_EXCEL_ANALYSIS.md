# Импорт коллекции из Excel — анализ и варианты

Документ описывает возможность загрузки Excel-файла пользователем для массового добавления монет в коллекцию. Реализация отложена.

---

## 1. Главная сложность — сопоставление (matching)

В каталоге монеты идентифицируются по внутреннему `id` (например `3637`, `4059`). В Excel пользователь может использовать:

| Что может быть в Excel | Наш каталог | Как сопоставить |
|------------------------|-------------|------------------|
| Номер ЦБ РФ (5111-0001, 3213-0005) | `id` — свой внутренний номер | Нужна таблица mapping catalog_number → coin_id |
| Название («Один червонец Сеятель 1975») | `title` | Fuzzy match по title, год, двор — ненадёжно |
| Год + номинал | `year`, `faceValue` | Слишком много совпадений |
| Свой нумератор | — | Не сопоставить |

**Вывод:** без единого идентификатора импорт будет работать плохо. Нужно либо договориться о формате (колонка с `coin_id`), либо иметь mapping каталожных номеров ЦБ РФ.

---

## 2. Варианты реализации

### Вариант A: «Строгий» — только колонка с ID

- Пользователь загружает Excel, где в колонке A — `coin_id` (наш id: 3637, 4059 и т.п.).
- Плюсы: простой парсинг, надёжный импорт.
- Минусы: пользователь должен знать наши id или экспортировать шаблон с id.

### Вариант B: Поддержка каталожных номеров ЦБ РФ

- Если в БД/каталоге есть `catalog_number` (5111-0001, 3213-0005):
  - Строим mapping catalog_number → coin_id.
  - В Excel: колонка с каталожным номером.
- Плюсы: формат привычен нумизматам.
- Минусы: нужно проверить, есть ли `catalog_number` в данных.

### Вариант C: Нечёткий поиск по названию/году

- Excel: название, год, номинал.
- На бэкенде: поиск по каталогу (fuzzy match).
- Плюсы: гибкость.
- Минусы: неоднозначности, ошибки, сложная логика.

### Вариант D: Шаблон Excel от нас

- Мы даём пользователю шаблон с колонками: `coin_id` (или `catalog_number`), `quantity`, `buy_price`.
- Пользователь заполняет и загружает.
- Плюсы: предсказуемый формат, меньше несовпадений.
- Минусы: пользователь должен использовать наш шаблон.

## 3. База данных и Supabase

**Текущая схема `user_collection`:**
```sql
user_collection (user_id, coin_id) unique(user_id, coin_id)
```

**При импорте:**
- Для каждой монеты: `INSERT` с `ON CONFLICT (user_id, coin_id) DO NOTHING` (или `DO UPDATE` при upsert).
- При ~500 монетах: один batch-insert лучше, чем 500 одиночных.

**Ограничения Supabase:**
- Free plan: до ~500 записей в одном insert обычно ок.
- RLS: только свои записи.
- Rate limits — при batch-insert нагрузка минимальна.

**Рекомендация:** один запрос с массивом строк (до 200–500 за раз), при большом файле — батчами.

---

## 4. Размер и валидация файла

| Параметр | Рекомендация | Причина |
|----------|--------------|---------|
| Макс. строк | 500–1000 | Лимит батчей, UX |
| Макс. размер | 1–2 МБ | Защита от перегрузки |
| Формат | .xlsx (можно .csv) | Обработка xlsx на сервере |

- Ограничить число добавляемых монет за один импорт.
- Проверять типы ячеек, пустые строки.
- Не выполнять макросы — только читать данные.

---

## 5. Архитектура: где выполнять логику

### Вариант 1: Клиент (браузер)

- Загрузка файла → парсинг (библиотека `xlsx` / sheetjs) → список `coin_id` → вызовы `addToCollection` или один batch через API.

Плюсы: не нужен серверный код для Excel.  
Минусы: ограничения Supabase по количеству запросов, возможные таймауты при большом файле.

### Вариант 2: Edge/Serverless

- API-маршрут (Vercel Function и т.п.): принимает файл, парсит, делает batch insert в Supabase.

Плюсы: контроль нагрузки, валидация, батчи.  
Минусы: при статическом экспорте (`output: 'export'`) API не попадает в прод — нужен отдельный backend.

### Вариант 3: Гибрид (клиент + RPC)

- Клиент парсит Excel → формирует массив `coin_id`.
- Один вызов Supabase RPC `import_collection(coin_ids text[])`, которая делает batch insert с `ON CONFLICT DO NOTHING`.

Плюсы: один запрос в БД, логика на стороне Supabase.  
Минусы: нужна настройка RPC в Supabase.

---

## 6. Вес и лимиты

**Хранение:**
- Одна строка в `user_collection`: ~50 байт.
- Импорт 500 монет: +25 КБ.
- При среднем портфолио 200–300 монет — рост незначительный.

**Лимиты по тарифу:**
- В `docs/ТАРИФЫ.md` описаны лимиты (например 50 монет на бесплатном).
- При импорте проверять: текущее количество + импорт не превышают лимит (если он реализован).

## 7. UX-поток (предварительный)

1. Страница «Моя коллекция» → кнопка «Импорт из Excel».
2. Выбор файла или drag-and-drop.
3. Парсинг и превью: какие монеты будут добавлены, сколько не найдено.
4. Пользователь подтверждает импорт.
5. Выполнение batch insert.
6. Отчёт: добавлено N, пропущено M (дубликаты), не найдено K (с указанием строк).

---

## 8. Итоговые шаги для реализации

1. **Определить формат matching:** только наш `coin_id` или `catalog_number` (если есть в данных).
2. **Выбрать архитектуру:** клиент + RPC или отдельный backend.
3. **Шаблон Excel:** выдать пользователю шаблон с колонкой id/catalog_number.
4. **Лимит на импорт:** 300–500 монет за раз.
5. **Supabase RPC** `import_collection(coin_ids[])` или batch insert.
6. **Фронт:** загрузка файла, парсинг (xlsx), вызов RPC, отображение результата.

---

## 9. Минимальный MVP

- Шаблон Excel с колонкой `coin_id`.
- Страница импорта: загрузка файла → парсинг на клиенте → batch insert через Supabase (RPC или несколько insert батчами).
- Отчёт: добавлено X, не найдено Y (с указанием строк).

---

## 10. Дополнительно проверить

- Есть ли в БД/каталоге поле `catalog_number` (номер ЦБ РФ) для mapping.
- Формат каталожных номеров в каталоге ЦБ РФ и в наших данных.
